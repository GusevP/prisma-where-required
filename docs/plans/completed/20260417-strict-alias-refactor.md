# Strict-alias refactor (prisma-where-required v0.2.0)

## Overview

Re-architect the generator to stop mutating Prisma's own `{Model}WhereInput` types and instead emit a sibling `{Model}WhereInputStrict` that carries the required-field constraint. This unblocks three user-visible improvements:

1. **Fixes a tenant leak**: nested relation args (`include: { posts: { where: {} } }`, `_count.select.posts.where`, relation filters like `posts: { some: {…} }`) currently bypass the required-field check because the convertor only touches types whose name starts with the current model.
2. **Removes OR/AND/NOT DX pain**: in-place mutation makes required-ness viral across `AND/OR/NOT` branches. With a sibling `WhereInputStrict`, combinators continue to reference permissive `WhereInput`, so the `{ organizationId: undefined }` workaround disappears from inside combinators.
3. **Adds schema-wide config**: `requiredFields = ["organizationId"]` generator option auto-applies to every model with that scalar, additive with `/// @where-required`.

Aggressive relation-filter scoping (`is/isNot/some/every/none` and XOR-embedded WhereInput refs *in relation-filter positions*) is included — the multi-tenant boundary applies through relation filters too. XOR positions in other contexts (nested update payloads, WhereUniqueInput discriminators) are **not** rewritten — see Non-goals.

### Non-goals (surfaces deliberately NOT rewritten)

| Surface | Reason |
|---|---|
| `{Model}WhereUniqueInput` | Unique lookups are primary-key-scoped by definition; re-requiring tenant field would force redundant filtering. Behavior matches v0.1. |
| `{Model}ScalarWhereWithAggregatesInput` (`groupBy.having`) | Operates on already-grouped aggregates; the outer `where` is strict, so tenant is already scoped. |
| `{Model}ScalarWhereInput` (nested update payloads) | Used inside `updateMany` nested updates; not a common multi-tenant-leak path. Re-visit if a user reports a leak. |
| `cursor` (uses `WhereUniqueInput`) | Same as WhereUniqueInput. |
| `data` (create/update payloads) | Not a filter surface. |

## Context (from discovery)

- **Entrypoint**: `src/generator.ts` (~75 lines) — reads DMMF, finds `prisma-client` output dir via `options.otherGenerators`, walks per-model files, calls `convert()` per model.
- **Convertor**: `src/convertor.ts` (~130 lines) — three ts-morph passes:
  - `removeQuestionTokenFromTargetFieldProperty`: mutates `{Model}WhereInput` fields (adds `| undefined`, drops `?`).

  - `removeQuestionTokenFromActionFuncProperty`: drops `?` from `{Model}Delegate` method params.
- **Per-model files**: Prisma 7's `prisma-client` generator emits `{output}/models/{Model}.ts` (one file per model, ~1100-1500 lines).
- **Relation-filter location**: `PostListRelationFilter` and `UserScalarRelationFilter` live in `models/Post.ts` / `models/User.ts` (not `commonInputTypes.ts`). The global walk over `models/*.ts` covers them.
- **XOR target positions** (to be exhaustively enumerated in Task 4 sub-step, based on the current 3-model fixture these already appear):
  - `PostWhereInput.author?: Prisma.XOR<Prisma.UserScalarRelationFilter, Prisma.UserWhereInput>` (line 235 in Post.ts)
  - `PostWhereUniqueInput.author?: Prisma.XOR<Prisma.UserScalarRelationFilter, Prisma.UserWhereInput>` (line 258) — **won't rewrite** (WhereUniqueInput is a non-goal)
  - Nested update/upsert payloads like `UserUpsertWithoutPostsInput` may contain `where: Prisma.UserWhereInput` — these are covered by Task 3's `where` rewrite, not by XOR rewrite.
- **Test harness**: `tests/type.test.ts` is auto-generated by `tests/generateTypeTestFile.ts`; currently hardcoded to `User.organizationId`. Runs via `npx prisma generate && tsc tests/*.test.ts --noEmit …`. Type-only, no runtime.
- **Schema fixture** (current): `prisma/schema.prisma` has User (`@where-required organizationId`), Post (`@where-required organizationId`), UserPost (has `organizationId` column, no annotation).
- **Schema fixture** (target — see Task 1):
  - `User`: keep `@where-required organizationId` → exercises annotation path.
  - `Post`: drop annotation, rely on `requiredFields` config → exercises config-only path.
  - `UserPost`: unchanged; under config `requiredFields = ["organizationId"]` it also becomes enforced → second config instance.
  - `Tag`: **new** — no tenant scalar, no annotation → true negative control (no Strict alias emitted).
  - `Memo`: **new** — `organizationId Int? /// @where-required` (nullable required field) + `owner User?` (nullable relation to a required model) → exercises `NullableScalarRelationFilter`, nullable-field-in-Strict-alias, and adds an inverse `memos Memo[]` on User (tests another `ListRelationFilter` path).

## Solution Overview

**Before (v0.1.x):** convertor mutates `{Model}WhereInput` in place — drops `?` and adds `| undefined` on target fields. Side-effects: (a) `AND/OR/NOT: UserWhereInput[]` inherits required-ness, forcing `{ organizationId: undefined }` in every branch; (b) cross-file refs (`User$postsArgs`, `UserCountOutputType…Args`) untouched because the walker excludes types starting with `${Model}$`.

**After (v0.2.0):**

1. Leave `{Model}WhereInput` unchanged. Prisma's own types stay pristine.
2. Emit `{Model}WhereInputStrict = Omit<{Model}WhereInput, 'f1'|'f2'> & { f1: <origType1> | undefined; f2: <origType2> | undefined }` alongside.
3. Rewrite **every** `where: (Prisma.)?{Model}WhereInput` across all model files → `…Strict`, drop `?`. Covers action args, delegate params (via args), nested include args, `_count` args, nested update/upsert `where`.
4. Rewrite `is/isNot/some/every/none: (Prisma.)?{Model}WhereInput` → `…Strict`, keep `?` (naturally optional). Also rewrite XOR-embedded WhereInput refs **only when inside a relation-filter property** (`author`-like fields on `{Model}WhereInput`), not inside WhereUniqueInput.
5. `AND/OR/NOT` branches continue referencing `{Model}WhereInput` (permissive) — OR/AND/NOT pain evaporates.
6. Generator accepts `requiredFields = [...]` config, combined additively with annotations.

## Technical Details

**Required-fields collection (per model):**

```
fields(Model) = annotated(Model) ∪ (scalar_fields(Model) ∩ config.requiredFields)
```

**Generator config parsing** (`options.generator.config.requiredFields`):

- Accept `string | string[]`, normalize to `string[]`.
- `trim()` each entry, drop empty strings, dedupe.
- After collection, if any configured field never matched any model's scalar, log a warning via `@prisma/internals` `logger.warn` (catches typos like `"organisationId"`).

**Strict alias emission**: read the original property type text from each required field's `PropertySignature` in `{Model}WhereInput` via ts-morph (`property.getTypeNode().getText()`); append a new exported `type {Model}WhereInputStrict = Omit<{Model}WhereInput, 'f1' | 'f2'> & { f1: <T1> | undefined; f2: <T2> | undefined }` alias in the same file.

- For nullable fields (`Int?`), the original type text is `… | number | null`; Strict becomes `… | number | null | undefined`. `Omit + &` keeps the property non-optional (required at type level) while allowing `null` and `undefined` values. Verify explicitly in tests.

**Rewrite matcher** (after alias emission, one global pass over all `models/*.ts`):

| Match | Action | Preserve `?` |
|---|---|---|
| `PropertySignature` named `where` with type `(Prisma\.)?{Model}WhereInput`, Model ∈ requiredSet | Rewrite type to `…Strict` | Drop |
| `PropertySignature` named `is`, `isNot`, `some`, `every`, `none` with same type match | Rewrite type to `…Strict` | Keep |
| `PropertySignature` whose enclosing type alias is a `{Model}WhereInput` AND the property type is `(Prisma\.)?XOR<_, (Prisma\.)?{Target}WhereInput>` with Target ∈ requiredSet | Rewrite second XOR arg to `…Strict` | Keep |

**XOR matcher gating**: the third rule is deliberately gated on the enclosing alias being `{Model}WhereInput` (not `{Model}WhereUniqueInput`, not nested update payloads). This matches the Post.author-style relation-filter case and nothing else.

**ts-morph Project strategy**: open a new `Project` per file (matches current `convert()` behavior). Rewrites don't need cross-file resolution — type text matching is purely textual.

**Delegate interface**: keep existing `?` removal on `{Model}Delegate` method params. Args types are already strictified via the type-reference rewrite, so the delegate inherits strictness transitively.

**Migration notes for downstream consumers** (include in release notes):
- Existing `/// @where-required` annotations work unchanged.
- Error messages now reference `{Model}WhereInputStrict` instead of `{Model}WhereInput`.
- Code with `{ organizationId: undefined }` inside `OR`/`AND`/`NOT` arrays can drop the workaround (still compiles if left in).
- Nested `include: { relation: { where: {…} } }` and `_count.select.{relation}.where` now enforce required fields. Consumers may see new errors where queries previously crossed tenants silently — **this is the intended tenant-safety fix**.
- Relation filters (`some/every/none/is/isNot` and XOR-direct shorthand `{ author: {…} }`) also enforce required fields.
- If your tsconfig has `exactOptionalPropertyTypes: true`, you must pass the required field explicitly (use `{ organizationId: undefined }` instead of omitting it).
- Unique ops (`findUnique`, `update`, `delete`, `upsert`, `cursor`, `groupBy.having`) are deliberately unchanged — see the Non-goals table in the README.
